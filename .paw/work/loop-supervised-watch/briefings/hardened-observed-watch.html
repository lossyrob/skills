Hardened Observed Watch

A simpler reliability design for long-running loop waits: keep the existing durable worker plus attached/backgrounded waiter, then harden the contracts that actually failed.

No v1 supervisor process Copilot CLI waiter stays attached Long watches use unbounded worker timeout Unknown liveness wakes loudly

Core decision

V1 unit: one run directory, one detached worker, one observed waiter.
Deferred: separate supervisor, generations, watch root, and new observed script family.
Reason: the known RCAs are contract failures in existing pieces, not proof that another long-lived process is required.

Centerpiece architecture: tie-in to Copilot CLI

Durability stays simple: one run directory remains the source of truth.
Wakeup stays explicit: Copilot CLI owns the attached waiter and sends shell_completed.
Robustness comes from contracts: unbounded watch intent, fallback, structured wakeups, and safe cleanup.

Supervisor cleanupSimplified
Use explicit cleanup command plus agent TODO.		Cleanup should be conservative, inspectable, and visible as workflow work, never a hidden side effect of waiting.

V1 hardening package

1
Missing-helper fallback
Waiter reads durable files directly and emits JSON instead of bare text failure.
2
Watch-until-terminal intent
-WatchUntilTerminal records intent and defaults worker timeout to unbounded.
3
Structured wakeups
lastWake preserves domain reason, exit code, summary, attempt, timestamp, and event path.
4
Safe cleanup
Invoke-LoopCleanup.ps1 reports by default, deletes only with -Apply, and is tracked by an agent cleanup TODO.

Reliability boundaries

The loop can make solidStill outside loop control
Durable state, liveness classification, structured wakeups, cleanup safety.Machine sleep, host process death, network outages, expired auth, wrong domain check logic.
Long watches do not end by omitted timeout choices when -WatchUntilTerminal is used.A caller explicitly choosing a finite timeout for a long watch.
Waiter remains observable when backgrounded through Copilot CLI.A waiter shell-detached by external scripts or user action.
Practical standard: fail loudly with recoverable state rather than attempting silent self-healing through another autonomous process.

Who runs cleanup?

Agent obligation
Create a loop-cleanup:<id> TODO when starting or finishing a long detached watch.
Two-step command
Run Invoke-LoopCleanup.ps1 -WhatIf, inspect it, then use -Apply only for safe eligible runs.
Done criteria
Deleted safe final-success runs, found nothing eligible, or recorded an intentional skip for ambiguous/useful runs.
Not automatic: worker and waiter never delete as a side effect. The agent is instructed to make cleanup visible and complete before final response or PR handoff.
